# A/B 面广告远程配置 — 流程图

> 仅 `ad_config_a` / `ad_config_b` · 树形简图  
> `isModeB` 来源见 [ab面判断逻辑](../ab面判断逻辑/流程图.md)。

---

## 1. 总览：fetch → 按面应用（先本地，后远程覆盖）

```
冷启动（AppAdsBootstrap）
  │
  ├─ hotResumeFastPath / 已 bootstrap?  → 跳过
  │
  ├─【拉取】FirebaseRemoteConfigFetcher.fetchAndActivate
  │    ├─ 成功 → 激活服务端 RC（含 ad_config_a、b）
  │    └─ 失败/超时 → 使用上次已激活缓存；applyByMode 仍执行
  │
  ├─【前置】isModeB 已确定（Mode2 + Debug 覆盖）
  │
  └─【广告】AdRemoteConfigBridge.applyByMode(context, isModeB)
        ├─ ① 先 apply 本地 assets JSON
        └─ ② 远程 ad_config_* 有效则覆盖 ①
```

---

## 2. fetchAndActivate

```
fetchAndActivate(interval)
  │
  ├─ setConfigSettingsAsync
  │    └─ Debug interval=0 / Release=3600
  │
  └─ fetchAndActivate()
         ├─ success → getString(ad_config_*) 可读最新远程值
         └─ fail    → 仍可读已激活缓存；远程可能为空 → 最终保持本地
```

> bootstrap 外包 `withTimeoutOrNull(8000)`；超时后仍 `applyByMode`（本地先行）。

---

## 3. applyByMode(isModeB) — 核心（先本地，后远程覆盖）

```
AdRemoteConfigBridge.applyByMode(context, isModeB)
  │
  ├─ 选定 key / assets 候选（A 面或 B 面）
  │
  ├─ ① applyLocalAssets
  │     ├─ 按顺序读 ad_remote_config_default_*.json
  │     ├─ parse 成功 → AdRemoteConfigManager.apply(本地)
  │     └─ 均未命中 → apply(parse(null)) 内置兜底
  │
  └─ ② applyRemoteFirebase
        ├─ getString(ad_config_a|b)
        ├─ 非空且 parse 成功 → apply(远程) 【覆盖 ①】
        └─ 空或 parse 失败 → 保留 ①，日志「保持本地/assets」
```

---

## 4. 与旧逻辑对比

| 场景 | 旧（远程优先） | 新（先本地后覆盖） |
|------|----------------|-------------------|
| 远程有效 | 仅远程 | 先本地 → 远程覆盖（最终=远程） |
| 远程空 | 才读 assets | 先本地已生效，远程跳过 |
| 远程非法 JSON | assets | 本地已生效，保留本地 |
| fetch 超时 | assets | 本地已生效；远程若有缓存且有效可覆盖 |

---

## 5. 配置来源（最终态）

```
最终配置 =
  远程 Firebase（非空且可 parse）  // 覆盖
  否则 本地 assets
  否则 parse(null) 内置兜底
```

A/B 各用独立 key：`ad_config_a` / `ad_config_b`。

---

## 6. 与 bootstrap 其它步骤

```
AppAdsBootstrap.run()
  ├─ fetchAndActivate          ← 本 skill
  ├─ Mode2Utils → isModeB      ← ab面判断逻辑
  ├─ AdRemoteConfigBridge      ← 先本地后远程覆盖
  └─ NotificationRemoteConfig… ← 不在本 skill
```
