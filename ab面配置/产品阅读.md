# A/B 面 — 产品阅读

## 用户看到什么

| 面 | 体验 |
|----|------|
| A 面 | 少广告、偏审核 |
| B 面 | 语言/引导/首页等更多广告 |

通知强弱取决于项目是否接 **通知 JSON 模块**（videodownload 有；PDF 金样未接）。

## 运营怎么控

| 手段 | 效果 |
|------|------|
| 总开关 =1 / =0 | 全员 B / 全员 A（自然 B 锁定例外） |
| 子项开关 | 控制是否检查买量来源、是否必须 GP 安装 |
| 广告 JSON `ad_config_a/b` | 各面广告位配置，与「是不是 B 面」分开 |

## 弱网会怎样

- FC 单次 8 秒超时，PDF 金样**最多再试 2 次**；仍失败用本地 A assets + 上次远程缓存判面
- B 面 commit 后远程 JSON 未到：后台**自动补拉**，避免 B 用户长期用 A 方案
- 安装来源 6 秒不确定：先当 A，**15 分钟内后台可能升为 B**

## 中途升 B（用户能感知什么）

| 时刻 | 用户看到 |
|------|----------|
| 先进语言页，后台仍判 A | 无 B 专属广告（正常） |
| 15min 内拉到买量来源升 B | **仍在语言页/主页时**应陆续出现原生、Banner 等（需 App 接 listener） |
| 升 B 但远程 B JSON 未到 | 仍像 A 广告方案，补拉/FC 刷新后应恢复 |
| 首次就是 B 且 commit 早于进页 | 进页即有 B 广告 |

**产品预期**：买量用户不应因为「判面比进页慢几秒」而整次启动都看不到语言原生/ Banner——这是 **必接** 能力，不是优化项。

## 和「本地 json」的关系

- 本地 json：**广告方案**的保底，启动很早就加载（A 面 assets）
- **是不是 B 面**：看买量来源、GP、远程总开关，**不是**本地 json 里写一个开关
